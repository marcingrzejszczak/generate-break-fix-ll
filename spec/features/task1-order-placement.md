# TASK 1: Order Placement System

## Overview

A two-service online ordering system where a customer places an order and the system
processes a payment before confirming the purchase.

## Services

### Order Service

- Accepts order requests from clients
- Listens on **port 3456**
- Communicates with the Payment Service through a network proxy

### Payment Service

- Authorizes payments for orders
- Listens on **port 8080**

### Network Proxy

- Sits between Order Service and Payment Service
- Listens on **port 9753**
- Forwards traffic from Order Service to Payment Service
- Can simulate network conditions (latency, timeouts, packet loss)

## Behavior

### Place an Order

**Trigger:** Client sends `POST /orders` to the Order Service

**Input:**
```json
{
  "productId": "<string>",
  "amount": <number>
}
```

**Flow:**
1. Order Service receives the request and generates a unique order ID
2. Order Service sends a payment authorization request to the Payment Service
   (via the proxy) with the order ID and amount
3. Payment Service authorizes the payment and returns a payment ID and status
4. If payment status is `AUTHORIZED`, the order status is `CONFIRMED`
5. If payment status is anything else, the order status is `FAILED`

**Successful Response (HTTP 201):**
```json
{
  "orderId": "<uuid>",
  "productId": "<string>",
  "amount": <number>,
  "orderStatus": "CONFIRMED",
  "paymentId": "<string>"
}
```

**Response includes a `Location` header** pointing to `/orders/{orderId}`.

### Payment Authorization

**Trigger:** Order Service sends `POST /payments` to the Payment Service

**Input:**
```json
{
  "orderId": "<string>",
  "amount": <number>
}
```

**Response (HTTP 200):**
```json
{
  "paymentId": "<string>",
  "orderId": "<string>",
  "amount": <number>,
  "status": "AUTHORIZED"
}
```

**Error (HTTP 400):** Returned when the request body is malformed.

## Constraints

- The Order Service MUST call the Payment Service through the proxy (port 9753), never
  directly (port 8080)
- Order IDs are UUIDs
- Payment IDs are generated by the Payment Service
- No persistence — orders and payments are not stored

## Failure Modes (unhandled in TASK 1)

- If the Payment Service is slow, the Order Service blocks indefinitely
- If the Payment Service is down, the Order Service returns a 500 error
- No timeout, no circuit breaker, no fallback
- Under sustained load with downstream latency, the Order Service exhausts its thread
  pool and becomes fully unresponsive
